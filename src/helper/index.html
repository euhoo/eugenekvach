Контекст и ограничения

Подключить LLM API было самой простой частью.

Сложность началась там, где свободное поведение модели встретилось с персональными данными, структурированной записью и production-ожиданиями.

  • Tool call не гарантирован.Модель может ответить прозой вместо вызова нужного инструмента.
  • История может устареть.Старый диалог способен подменить свежий результат сервиса.
  • Динамика ломает cache.Изменение prefix повышает latency и стоимость input-токенов.
  • Память накапливает мусор.Без lifecycle появляются дубли и противоречия.
  • Unit-теста недостаточно.Нужно проверять диалог, routing, tools и запись в БД вместе.
  • Данные нужно разделять.Пользовательские контуры не должны пересекаться.

Decision 01 Надёжность до LLM

Критичный интерфейс нельзя гарантировать одним prompt.

Что значит deterministic fast-path?

Для известного intent результат задаёт код, а не выбор модели. Общие формулировки по-прежнему уходят в LLM-контур.

Проблема

Модель иногда отвечала текстом вместо ожидаемого интерфейса с кнопками. Полный LLM-путь занимал около 2575 мс.

Решение

Статичные capability-триггеры вынесены в regex fast-path, который выполняется до вызова модели.

Результат

~31 мс · ×80 · 0 LLM calls

Trade-off

Regex не пытается понимать всё. Любая непокрытая формулировка продолжает идти через общий LLM-путь.

Decision 02 Cache-first prompts

Сократить prompt недостаточно. Нужно сохранить стабильный prefix.

Что такое prefix cache?

Провайдер повторно использует вычисления для неизменной начальной части prompt. Поэтому стабильные правила идут первыми, а динамический контекст — после них.

Проблема

Динамические блоки меняли начало prompt и снижали cache hit. Механическое сокращение могло удалить load-bearing правила.

Решение

Static и dynamic части разделены; system prompts сжаты только после проверки критичных правил поведения.

Результат

36% → 50% cache hit · около −22% input cost

Дополнительно

Medical-agent prompt уменьшен на 49%, main-agent prompt — на 47%.

Decision 03 Поведение как контракт

Тест функции не доказывает, что работает пользовательский сценарий.

Что проверяет scenario harness?

Не отдельную функцию, а цепочку: Telegram-диалог, routing, tool calls, ответ пользователю и итоговое изменение данных.

Проблема

Unit-тесты не ловили регрессии полного Telegram-диалога: routing, tool use, ответ и запись в PostgreSQL.

Решение

Созданы Telegram-mock harness, сценарии на реальной тестовой БД и regression/golden cases из production-ошибок.

Результат

2454 теста разных уровней · 277 suites · 2 бага до релиза

Граница

Набор смешанный: unit, integration, DB и scenario tests. Это не 2454 unit-теста.

Decision 04 Memory lifecycle

Vector search не превращает накопленные записи в память.

Зачем памяти два слоя?

Короткий контекст должен устаревать, а долгосрочные факты — проходить recall, нормализацию и дедупликацию. У них разный lifecycle.

Working memory

Временный контекст хранится с TTL и счётчиками обращений.

Long-term memory

PostgreSQL/pgvector отвечает за recall, нормализацию и дедупликацию фактов.

Cleanup

24 записи → 15 уникальных фактов

Вывод

Память требует отдельного data lifecycle и наблюдаемости, а не только embeddings.

Production и эксплуатация

Система заканчивается не на ответе модели.

Отдельные контуры отвечают за доставку, ограничения и восстановление, не маскируя программные ошибки.

  • RuntimeTypeScript / Node.js core + Telegram transport
  • Data layerPostgreSQL / pgvector
  • Providersmulti-provider LLM API через Vercel AI SDK
  • DeliveryGitHub Actions + self-hosted runner
  • Limitsтокены, ASR-минуты и стоимость
  • Resilienceизвестные transient network errors без blanket-ignore

Что изменилось в подходе

Четыре вывода, которые переношу в следующие AI-системы.

  1. Prompt не является гарантией.Критичные пути требуют deterministic guards.
  2. Evals проверяют последствия.Ответ модели, tool call, запись в БД и полный сценарий.
  3. Память — отдельная подсистема.Нормализация, дедупликация и наблюдаемость обязательны.
  4. Следующий шаг — hybrid contour.Дальнейший deterministic write path и трассировка памяти.
