Состав маршрута

Не модель, а система вокруг неё.

Каждый слой спроектирован под свой класс отказов и проверен в production — от routing до стоимости input-токена.

  • Multi-agent orchestration Coordinator определяет путь запроса; специализированные агенты получают узкую зону ответственности, а не один универсальный prompt.
  • Tool use Function calling к domain-сервисам: чтение и структурированная запись вместо свободного текста в ответе.
  • Двухслойная память Working memory хранит контекст с TTL. Long-term отвечает за recall, нормализацию и дедупликацию фактов.
  • Детерминированный контур Критичные триггеры выполняются до модели: ~31 мс вместо ~2575 мс и ноль LLM-вызовов на известном пути.
  • Scenario evals Проверяется не «хороший ответ», а цепочка: диалог → routing → tool call → запись в БД. Сценарии строятся из реальных ошибок.
  • Стоимость и latency Cache-first prompts и разделение static/dynamic частей сохраняют prefix cache и снижают стоимость input-токенов.

Правила маршрута

Четыре правила, которые вынес из production.

  1. 01
    Начинать с failure mode, а не с модели. Что может сломаться — галлюцинация tool call, устаревшая история, деградация cache — то и формирует архитектуру.
  2. 02
    Детерминированно там, где можно. Модель дорогая и недетерминированная. Код до модели — быстрый, одинаковый и проверяемый.
  3. 03
    Сначала измерить, потом оптимизировать. Latency, стоимость input-токенов и качество сценариев — метрики первого класса, а не догадки «на глаз».
  4. 04
    Поведение — тестируемый контракт. Сценарий, воспроизводящий реальную ошибку, ценнее unit-теста на отдельную функцию.
